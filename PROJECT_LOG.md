- Designed the Feature 006 incremental architecture, design only, no code written yet. An initial proposal (a growing sharelog.state.json archive of every parsed historical share) was rejected: it would duplicate the complete share history, grow without bound, require rewriting the entire archive on every run, and leave all statistics computation O(total historical shares).
- Approved design instead: analytics_state.json will contain a version and generation counter; per-file fingerprints and byte offsets; per-file cumulative pool/user/worker partial accumulators; the accepted-sdiff values required for exact percentiles; and bounded recent tuples retained for 48 hours (not full share records) used only to rebuild rolling windows and daily_bests/live_ticker.
- Decided per-file partial state is an intentional architectural choice, not an implementation detail: it is what makes a truncation/replacement reset local to one file (discard and rebuild just that file's partial) rather than requiring either global subtraction (not possible without per-file attribution anyway) or a full-pool rebuild on every isolated reset. It is also what allows a deleted or rotated source file's statistics to be preserved indefinitely: its partial remains in analytics_state.json and continues contributing to merged totals even after the file itself disappears from disk.
- Decided the replacement/truncation fingerprint is device, inode, size, mtime_ns (nanosecond resolution), the stored byte offset, and a SHA-256 hash of the first min(offset, 65536) bytes of the file. A file is treated as consistent (safe to read incrementally from the stored offset) only if dev/ino match, current size is not less than stored size, current mtime_ns is not less than stored mtime_ns, and the prefix hash still matches; any mismatch triggers a reset (that file's partial and recent tuples discarded, offset reset to 0, re-read from the start). A path that reappears after its file was absent from a scan is validated through this exact same check before being treated as a continuation of the old file — no special-casing, since the fingerprint check already distinguishes "same file, was briefly inaccessible" from "different file reusing an old path."
- Decided analytics_state.json is the single, versioned, atomically-written (temp file plus os.replace) durability point for both cursor and analytics state, chosen specifically to avoid the two-separate-files-can-disagree failure mode a split cursor-file/archive-file design would have. analytics.json remains a separately written public output; it may be up to one run stale if the process crashes between the two writes, but must never be partially written, which the existing atomic write already guarantees.
- Decided the per-run transaction sequence: for each currently-present sharelog file, run the fingerprint consistency check, read new complete lines only (a trailing partial line is left unconsumed for a future run, so a line split across two runs is never double-processed or lost), update that file's in-memory partial, then prune every file's recent-tuple list to the last 48 hours. Only after all files are processed in memory is analytics_state.json written atomically — this write is the single commit point. Only after that write succeeds is analytics.json recomputed from the now-durable state and written atomically. A crash before the state write means nothing new was committed, so the next run safely reprocesses the same unconsumed bytes; a crash after the state write but before the analytics.json write leaves analytics.json merely stale by one run, with no reprocessing and no risk of duplicate counting, since the cursor advance was already durable.
- Decided exact percentile semantics remain unchanged and no approximation (e.g. t-digest or similar streaming quantile structure) is permitted. Chose to retain the accepted-sdiff distributions needed for exact percentiles, scoped per file per entity (pool/user/worker) rather than as one global list, specifically because per-file scoping is what makes a truncation reset a local discard-and-rebuild rather than requiring value-level subtraction from a shared list. The existing percentile()/median()/_BestTracker primitives from pool_statistics.py are reused unmodified for the merge step; compute_pool_statistics/compute_user_statistics/compute_worker_statistics themselves remain unmodified but are no longer called by the live incremental path, and are instead retained as the reference implementation for a required differential test proving the incremental merge produces identical output to the original batch computation over the same data.
- Decided parse_share_analytics.py remains unmodified; the new incremental-read module implements its own offset-aware line reader duplicating the existing decode/skip-bad-line/non-dict-JSON pattern, rather than extending the shipped file, since offset-aware reading was not something the shipped reader needed to support before now.
- Decided analytics_builder.py's build_analytics() will no longer call compute_pool_statistics/compute_user_statistics/compute_worker_statistics/compute_temporal_analytics directly; it will call new merge functions operating over analytics_state.json's partials instead of a raw share list. All three Majors were subtle correctness/robustness gaps in the durability layer itself, not data-dependent edge cases, and none were exercised by current production data (largest real sharelog is ~22.5KB) — found only through deliberate adversarial construction:
  1. The prefix-hash consistency check capped verification at a fixed 65536 bytes regardless of a file's true consumed offset; content tampering beyond that cap, within an already-large committed region, went permanently undetected as long as size/mtime were preserved — the same tampering shape as an earlier-fixed bug, just deeper into a larger file.   2. Fixed by wrapping the consistency check and both reads in the same try/except OSError, so a mid-scan vanish skips just that one file and the run continues normally for every other file.

A file whose latest bytes form a persistent, never-completed trailing line (the realistic steady state of an actively-tailed sharelog) never satisfied the "skip the write" fast path's exact offset-equals-size condition, so every run fell through to the full per-file branch and unconditionally marked the run as changed, re-introducing the exact rewrite-every-run cost the earlier fix for this eliminated.
