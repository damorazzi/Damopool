## 2026-07-23 (Milestone 31)

- **Human brief:** Worker Session Accepted/Rejected Counts -- each worker's CURRENT CONNECTION SESSION accepted/rejected share count, additive alongside (never replacing) the existing lifetime `accepted_count`/`rejected_count`. Worker-scope only (pool/user stay lifetime -- no coherent "session" concept exists at those scopes, per Human decision). No inactivity timeout; a session resets only when the underlying connection genuinely changes.
- **Investigation, round 1:** the original mechanism sketch (track the highest `clientid` seen per workername) was re-verified against real data and found broken: CKPool's connector explicitly recycles `clientid` (a small connection-slot integer -- confirmed via `ckpool.log`'s own "Connector recycling client N" line), including reuse across entirely different, non-overlapping connections. A refined "any clientid change = new session" rule was also found broken by a genuinely concurrent-connection workername (multiple simultaneous clientids under one workername, confirmed via CKPool's own native `pool.status` reporting `"workers": 4` for that exact case).
- **Investigation, round 2 (deeper signal search):** re-examined the full set of fields present on a real `.sharelog` record and found `enonce1` (the stratum extranonce1) -- previously and incorrectly dismissed as "just a re-encoding of clientid" -- is in fact the correct signal. Checked across every record in this pool's full history (195k+ shares): `clientid` maps to *multiple* `enonce1` values over time (confirmed: `clientid=7` alone had three distinct `enonce1` values, one per genuinely separate connection, including two different users at different times), while `enonce1` never once mapped to more than one `clientid` or more than one workername, with 100% field coverage. This matches `enonce1`'s actual protocol purpose (partitioning each connection's own nonce search space to prevent hash collisions between concurrent miners) -- reuse would be a CKPool protocol bug, not merely inconvenient here.
- **Amendment approved:** session-boundary key is `(workername, enonce1)`, not `(workername, clientid)` -- `clientid` is retained internally per session (for potential future debugging value) but is never the disambiguating key. Disclosed, Human-acknowledged limitation: no mathematical proof `enonce1` can never repeat over an arbitrarily long pool lifetime, only that it is protocol-designed for uniqueness and has never repeated once in this pool's entire observed history; if it ever did, the failure mode is identical to (and far rarer than) naive clientid-keying's routine failure. No `ckpool.log` dependency at runtime and no inactivity timeout, per explicit Human instruction -- a quiet-then-resuming connection under its own known `enonce1` always continues, never resets.
- **Architecture:** new standalone module `worker_sessions.py`, same module-separation precedent as `histogram_builder.py`/`ckpool_native_stats.py`/`block_progress.py` -- `analytics_state.py` remains completely untouched. Unlike those forever-cumulative, order-*independent* per-file accumulators, session detection is inherently order-*dependent*: every run gathers all newly-read share tuples across every sharelog file touched into one batch, sorts each workername's own tuples by `createdate`, and replays against a single persistent global per-`(workername, enonce1)` state dict -- simpler than a per-file-partial-then-merge design, since that model would be actively wrong here. First-run full-history replay (correctly reconstructing every worker's true current session from this pool's entire history) falls out of this design for free, at the same "read sharelog bytes an extra time in exchange for genuine module isolation" cost already accepted for `histogram_builder.py` -- not a new one-off migration, superseding an earlier, less accurate "imperfect bootstrap" plan.
- **Schema:** additive only. `session_accepted_count`, `session_rejected_count`, `session_started_at` on `workers[...]` only. `schema_version` bumped `"1.4"` -> `"1.5"`.
- **Frontend:** Worker Detail gained two new stat tiles ("Session Accepted", "Session Rejected") alongside the unchanged existing lifetime "Accepted Shares"/"Rejected Shares" tiles (15 -> 17 tiles), plus a small "Current session since HH:MM" caption reusing the existing `.stat-tile__label` CSS rule rather than a new component or a separate large tile, per explicit Human instruction.
- **Process, corrected proactively this time:** having learned the Milestone 30 lesson, both Code Reviewer and Test Engineer were dispatched against the finished implementation *before* presenting this milestone as complete, with no approval sought first -- no repeat of the M30 process gap.

- Decided per-file partial state is an intentional architectural choice, not an implementation detail: it is what makes a truncation/replacement reset local to one file (discard and rebuild just that file's partial) rather than requiring either global subtraction (not possible without per-file attribution anyway) or a full-pool rebuild on every isolated reset. - Decided the replacement/truncation fingerprint is device, inode, size, mtime_ns (nanosecond resolution), the stored byte offset, and a SHA-256 hash of the first min(offset, 65536) bytes of the file. A file is treated as consistent (safe to read incrementally from the stored offset) only if dev/ino match, current size is not less than stored size, current mtime_ns is not less than stored mtime_ns, and the prefix hash still matches; any mismatch triggers a reset (that file's partial and recent tuples discarded, offset reset to 0, re-read from the start). A path that reappears after its file was absent from a scan is validated through this exact same check before being treated as a continuation of the old file — no special-casing, since the fingerprint check already distinguishes "same file, was briefly inaccessible" from "different file reusing an old path."

Only after all files are processed in memory is analytics_state.json written atomically — this write is the single commit point. Only after that write succeeds is analytics.json recomputed from the now-durable state and written atomically. A crash before the state write means nothing new was committed, so the next run safely reprocesses the same unconsumed bytes; a crash after the state write but before the analytics.json write leaves analytics.json merely stale by one run, with no reprocessing and no risk of duplicate counting, since the cursor advance was already durable.
